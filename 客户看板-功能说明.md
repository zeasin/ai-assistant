# 客户管理看板 — 页面功能说明

> 本文档描述 `/customers` 页面的功能、逻辑、数据来源和技术架构。

---

## 一、页面概述

客户管理看板是 AI 助手内置的 CRM 页面，访问路径 `/customers`。页面由以下几个区域组成：

- AI 客户分析报告
- 总览统计
- 客户漏斗（按阶段）
- 线索漏斗（按状态）
- 年度营收表
- 待办事项
- 最近活动
- 原始数据详情（可折叠）

---

## 二、数据来源

### 2.1 配置方式

在配置页面（`/config`）的「数据目录配置」卡片中设置「客户数据目录」。

该路径相对于笔记库根目录（`app.baseDir`），例如设置为 `客户管理/data` 时，实际读取路径为 `{baseDir}/客户管理/data/`。

### 2.2 数据读取机制

- 程序自动扫描配置目录下 **所有 `.json` 文件**
- 每个 JSON 文件的文件名（不含扩展名）作为数据分组名称
- 从每个文件中提取所有 **顶层数组** 的值作为数据条目
- 默认期望的文件：`客户数据.json`、`线索数据.json`、`跟进记录.json`

### 2.3 数据格式约定

每个 JSON 文件的顶层结构：
```json
{
  "meta": { "lastUpdated": "2026-05-29" },
  "customers": [ ... ],
  "pipelines": [
    { "stage": "意向", "color": "#7950f2" },
    ...
  ],
  "leadStatuses": [
    { "status": "待跟进", "color": "#fab005" },
    ...
  ]
}
```

数据条目为 `Map<String, Object>` 结构，支持任意字段。嵌套的 Map 和 List 在原始数据表中自动隐藏。

## 三、服务端逻辑（CustomerController）

### 3.1 数据加载流程

```
customersPage() 被调用
     │
     ├─ loadAllDataGroups() ── 扫描目录，读取所有 JSON 文件
     │       │
     │       └─ 返回 Map<分组名, List<数据条目>>
     │
     ├─ 从中提取 customers / leads / records
     │
     ├─ loadCustomerData() ── 读取 客户数据.json 获取元数据
     │       (pipelines, leadStatuses, meta.lastUpdated)
     │
     └─ 计算各项统计指标并注入 Model
```

### 3.2 统计指标计算

| 指标 | 计算逻辑 | Model Key |
|------|----------|-----------|
| 客户总数 | `customers.size()` | `total` |
| 活跃客户 | `stage != "已流失"` 的客户数 | `active_count` |
| 已流失 | `stage == "已流失"` 的客户数 | `churned_count` |
| 线索数 | `leads.size()` | `lead_count` |
| 已交付数 | stage 为已交付/进行中/已签约-待首款 | `delivered_count` |
| 签约总额 | 所有客户 `contractAmount` 之和 | `total_contract` |
| 收款总额 | 所有客户 `paidAmount` 之和 | `total_paid` |
| 年度目标 | 硬编码 500000 | `target` |
| 待办事项 | 有 `nextFollowUp` 或 notes 含"催"字的客户/线索 | `todos` |

### 3.3 漏斗计算

**客户漏斗**：按照 `客户数据.json` 中 `pipelines` 数组定义的阶段顺序，统计每个阶段的客户数量。每个阶段显示进度条（宽度 = 该阶段数量 / 最大阶段数量 × 100%）。

**线索漏斗**：按照 `leadStatuses` 数组定义的状态顺序，统计每个状态的线索数量。

### 3.4 其他逻辑

- **跟进记录分组**：按 `customerId` 将跟进记录分到对应的客户和线索
- **最近活动**：按日期排序取前 10 条跟进记录
- **营收客户**：`contractAmount > 0` 的客户列表
- **待办排序**：按 `nextFollowUp` 日期升序排列，最多显示 8 条

### 3.5 其他端点

| 端点 | 方法 | 功能 |
|------|------|------|
| `/customers/all` | GET | 所有客户列表页 |
| `/leads/all` | GET | 所有线索列表页 |
| `/api/customers/ai-analysis` | GET | SSE 流式 AI 客户分析 |
| `/api/customers/add` | POST | 新增客户 |
| `/api/leads/add` | POST | 新增线索 |
| `/api/records/add` | POST | 添加跟进记录 |

## 四、AI 客户分析

### 4.1 触发方式

- 页面加载时自动触发（非强制模式，使用缓存）
- 点击「重新分析」按钮强制刷新

### 4.2 缓存机制

- 当天首次分析后结果被缓存
- 再次访问直接返回缓存，不重复调用 AI
- 缓存 key 为日期字符串 `yyyy-MM-dd`

### 4.3 数据摘要构建

`buildDataSummary()` 方法动态读取所有 JSON 文件中的数据，构建为文本摘要。每条记录用 `key=value` 格式呈现，跳过嵌套对象和数组。

### 4.4 AI 提示词

提示词要求 AI 生成包含以下部分的报告：
1. 整体概况
2. 客户阶段分析
3. 重点客户识别
4. 跟进建议
5. 风险提示

### 4.5 SSE 流式响应

通过 Server-Sent Events 实现实时流式输出，事件类型：
- `status` — 分析状态提示
- `text` — AI 分析结果（Markdown 格式，前端用 marked.js 渲染）
- `error` — 错误信息
- `done` — 完成信号

## 五、原始数据展示

### 5.1 动态表格

原始数据区域放在 `<details>` 折叠面板中，内容是动态生成的：

- 遍历 `rawDataGroups`（`Map<分组名, List<数据条目>>`）
- 每个分组生成一个卡片，包含一个数据表格
- 表格列头取自第一条数据的 key，通过 `@utils.label(key, keyLabels)` 显示用户自定义标签
- 未配置标签的 key 显示原始英文

### 5.2 数据录入

每个分组对应一个「添加」按钮，点击打开通用添加弹窗。

添加方式：用户在文本框中用自然语言描述要添加的数据，点击提交后通过 `aiSend()` 发送给 AI，AI 理解后写入对应的 JSON 文件。

示例输入：
```
在客户数据中新增客户：姓名=张三，公司=ABC科技，阶段=意向
```

### 5.3 无数据引导

当客户数据目录未配置或目录为空时，页面顶部显示引导卡片，提示用户前往配置页面设置数据目录。
